                    # 一、背景

                    ​ 在622团队中，项目需要共同协作完成，因个人差异的存在，每个人都会有不同的代码风格和习惯。想要高效率协同完成任务，统一的规范是必不可少的。

                    # 二、目标

                    ​ 统一代码风格、命名规范，增强代码可读性和可维护性，提高团队协作的开发效率。

                    # 三、编程规范

                    【强制】必须遵守的规范

                    【推荐】希望遵守的规范

                    【禁止】禁止使用的规范

                    前提基础：模块化编程思想！

                    622实验室团队标识符：Stt（Six two two）

                    # 流程梳理

                    1. 【推荐】在开始编写代码之前，需要对实现逻辑十分清晰，建议在写代码之前先梳理流程并画出流程，一方面利于代码编写，另外一方面利于团队其他成员能够更快熟悉你所负责的部分。

                    # 变量、常量与宏

                    1. 【强制】整数类型使用C99中<stdint.h>定义的扩展整数类型intN_t 、uintN_t（比如uint8_t、uint16_t）。避免直接使用int。

                    2. 【强制】对于一些全局变量必须添加static进行修饰，不能直接在多个文件间相互使用，想要跨文件使用必须在提供函数接口（不能直接使用！！）。比如：

                      //wifi.c

static uint8_t s_flag;
uint8_t *Stt_App_WifiFlag_Get()
{
	return &s_flag;
}

void Stt_App_WifiFlag_Set(uint8_t flag)
{
	s_flag = flag;
	return;
}

                    # 函数

                    1. 【强制】模块内部调用的函数必须要加static防止外部调用。
                    2. 【强制】函数如果有参数必须需要对参数合法性进行检查和处理。
                    3. 【强制】函数错误码类型使用int32_t，返回C标准库<errno.h>的通用错误码或各模块内的专用错误码，避免直接返回数值，函数成功返回值为0
                    4. 【强制】函数编写完成后，若函数内部有文件IO操作，缓冲区申请操作等容易产生段错误的操作，需要反复检查保证有打开有关闭、有申请有释放
                    5. 【禁止】禁止使用超过1KB的局部变量。大数据量使用动态申请或者静态全局，避免堆栈溢出。
                    6. 【推荐】重复代码或者功能尽量整理成通用函数或功能函数。

                    # 头文件

                    1. 【强制】头文件用于声明该模块的对外接口和数据结构，通过头文件能了解该模块所有功能接口。
                    2. 【强制】使用#define防止多重包含，宏为文件夹名+文件名，所有字母大写，前后加双下划线__。
                    3. 【推荐】头文件引用排序即可划分开
                    #include <stdio.h>		//系统标准库

                    # 安全规范

                    1. 【强制】在动态申请内存时，无使用情况下必须释放

                    2. 【强制】在循环中若有申请内存操作，应该注意避免出现多次释放的操作

                    3. 【强制】在不满足条件或者发生异常错误函数需要return时，一定要注意return前是否有申请的资源未释放

                    4. 【强制】编写代码时时刻注意避免出现野指针

                    # 排版格式

                    1. 【强制】采用缩进风格，对齐使用Tab键，缩进为1个Tab。
                    2. 【强制】函数内相对独立的程序块加空行用于区分，函数间加空行用于区分。
                    3. 【强制】大括号单独换行。如下
                    if(XX)
{
	printf("hello\n");
}

                    # 注释格式

                    # 1.注释原则

                    项目开发中，尽量保持代码注释规范和统一。

                    注释方便了代码的阅读和维护。

                    边写代码边注释，修改代码时要相应修改注释，保证注释和代码的一致性。

                    注释要简洁明确，不要出现形容词。

                    通过注释可以快速知道所写函数的功能，返回值，参数的使用。

                    # 5.建议

                    一般情况下，源程序有效注释量必须在20％以上。 注释不宜太多、不宜太少，准确易懂简洁；

                    注释格式尽量统一，建议使用“/* …… */”；

                    避免在一行代码或表达式的中间插入注释；

                    说明：除非必要，不应在代码或表达中间插入注释， 否则容易使代码可理解性变差。

                    通过对函数或过程、变量、结构等正确的命名以及合理地组织代码的结构，使代码成为自注释的。

                    清晰 准确的函数、变量等的命名，可增加代码可读性，并减少不必要的注释。

                    在代码的功能、意图层次上进行注释，提供有用、额外的信息。

                    说 明：注释的目的是解释代码的目的、功能和采用的方法，提供代码以外的信息，帮助读者理解代码，防止没必要的重复注释信息。

                    # 调试

                    1. 【强制】所有嵌入式项目可以统一使用一个LOG模块，不同类别的调试打印信息需要有对应的LOG等级，每一条打印信息必须包含年月日时分秒毫秒。（LOG模块代码开发由各个项目组负责人参与撰写与维护迭代，各项目通用）
                    2. 【强制】调试信息要打印有用，能定位问题的信息，避免打印111、222这种无意义LOG。

                    # 代码审查

                    防线 责任人 负责工作
                    第一道防线 开发者本人 在功能点实现完成后，应当进行烧录自测，和代码的重新审查，逻辑重新梳理，防止出现BUG
                    第二道防线 项目组负责人或成员 在开发者本人经过第一道防线后，提交代码到临时分支，项目组负责人或成员需要对其代码进行二次审查，审查没问题后将代码合并到项目分支

                    # 项目代码合作开发建议

                    【推荐】举个例子：当项目中用到DHT11、GY302光敏传感器等以及ESP8266、LoRa通讯模块,需要通过通讯模块上传温湿度、光照数据。我们可以将其划分为驱动层、应用层，目录结构可以是如下

                    基于驱动层目录结构：

                    ​ device （驱动：提供底层接口）

                    ​ -> sensor

                    ​ stt_dev_dht11.c

                    ​ stt_dev_dht11.h

                    ​ stt_dev_gy302.c

                    ​ stt_dev_gy302.h

                    ​ -> communication

                    ​ stt_dev_esp8266.c

                    ​ stt_dev_esp8266.h

                    ​ stt_dev_lora.c

                    ​ stt_dev_lora.h

                    基于应用层：

                    ​ app （应用层：）

                    ​ -> data

                    ​ stt_app_data.c //提供获取各种所需形式数据的接口

                    ​ stt_app_data.h

                    ​ -> msg_handler

                    ​ stt_app_wifi.c //以WiFi为发送媒介的业务逻辑

                    ​ stt_app_wifi.h

                    ​ stt_app_lora.c //以Lora为发送媒介的业务逻辑

                    ​ stt_app_lora.h

                    ​ 【推荐】不管每个人负责什么模块，都应该时刻以调用者什么都不知道什么都不懂的心态去封装接口，其他人调用你的接口就能实现他所需要的功能，而不需要关心你的接口里面做了什么。
