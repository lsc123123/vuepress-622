              # 一、背景

              ​ 在622团队中，项目工程代码往往由多人共同协作开发，因此我们需要一个清晰的代码管理流程，能够很好的管理每个人的代码，也能够更好的提高实际开发效率。

              # 二、目标

              ​ 1、学习、熟练、掌握Git版本管理工具，养成良好的编程习惯

              ​ 2、建立622团队项目工程代码管理流程。

              # 三、什么是Git

              ​ git是个分布式 (opens new window)版本管理工具，它可以在任何时间点，将文档的状态作为更新记录保存起来，也可以在任何时间点，将更新记录恢复回来。在开发中，Git已成为现在主流的一种代码托管技术（版本管理工具），基本上大多数的公司都在使用Git进行协同开发。很多代码托管平台也是基于Git来实现的。

              # 四、Git的使用

              ​ 在622团队嵌入式项目开发，我们选用Git作为我们的版本管理工具，详细的命令使用这里不多叙述，网上资料比较齐全，自行百度学习。

              # 2.创建个人分支

              ​ 1、个人分支的用途：在项目代码开发的过程中，你可以将自己的代码修改提交到你的远程个人分支上（小改了一下但是并未完成某个功能点），但是一般远程个人分支用于提交代码（已经完成某个功能点，并自身已对代码进行审查）的时候。

              ​ 2、个人分支的创建：方法1：远程个人分支创建方式与项目分支一样，只是在创建的时候，Soure需要选择到你所负责的项目分支。

              ​ 方法2：无需在Github上创建，在本地push到远程时即可自动创建。

              # 六、Git Commit规范

              <提交类型>: <主题> 

 影响范围:
 修改描述:
 自测结果:

# 提交类型:（必填）
#   字段选择如下:
#     Feature：新功能（feature）
#     Fix：修复bug 
#     Refactor：重构
#     Perf: 性能改善
#     Chore: 其他修改, 杂项

#    Commit目的的简短描述，不少于5个字符，不超过50个字符, 即 What
# 修改描述(开头空两格):
#    对本次Commit的详细描述，可以分成多行, 即 Why and How，若修改简短，可忽略不写.
# 影响范围:（必填，开头空两格）
#      用于说明 修改影响的功能范围
# 自测结果(开头空两格)：
#    确认是否已自测，提交前需要本地进行编译、烧录、测试, !!!禁止未测试的代码提交!!!
#    根据实际情况，可填写：“已烧录自测”、“无法自测”、“自测通过”

              # 七、代码审查与提交流程

              ​ 【强制】在仓库中，主分支不允许有如何的代码提交操作，所有项目代码的开发，最后均上传合并的各项目组的项目分支。

              # 1.代码审查

              ​ 代码审查遵循以下流程规则：

              防线 责任人 负责工作
              第一道防线 开发者本人 在功能点实现完成后，应当进行烧录自测，和代码的重新审查，逻辑重新梳理，防止出现BUG
              第二道防线 项目组负责人或成员 在开发者本人经过第一道防线后，提交代码到临时分支，项目组负责人或成员需要对其代码进行二次审查，审查没问题后将代码合并到项目分支

              ​ 在某个功能节点的代码开发结束后，首先开发者本人要对代码进行重新审查，确保代码逻辑、代码规范等符合要求，在自审核结束后，进入到第二道审查。

              ​ 项目组负责人或成员通过开发者提供的diff，进行代码复审。（diff可以简单理解为代码的变化，上一次跟这一次的区别）

              # 2.代码提交

              ​ 在开发者完成代码审查后，将代码提交到自己的个人分支，提交Commit必须遵循Commit规范。在代码提交到个人分支后，提供diff链接到项目组负责人或者审查成员进行第二重审查，再三确认无误后，再将代码从个人分支合并到主分支（合并结束后，确保个人分支被删除）

              # 八、注意事项

              ​ 由于目前仓库未设定Commit规范的检验、合并规则、以及仓库权限规则等这些东西，所以一切操作请谨慎再谨慎，特别是代码提交、合并的步骤一定要正确，并且各团队成员不能随意删除仓库内的分支（个人分支除外）。

              ​ 还有一点也非常重要，就是备份。除了远程仓库的存储之外，各成员的本地仓库也要留有备份，不要随意删除。最好有专门团队成员负责定期对远程仓库重要版本代码进行本地备份。
